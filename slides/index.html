What I learned about Lore

HIPAA-regulated

Medicare ACO context. Eligibility data is PHI from the moment it lands. Every architectural choice has to pass that filter.

Partner-driven

Employers, brokers, payers, ACOs send members in via eligibility feeds. Broken feed = no member to enroll = no product.

08 / 27

Two ingest paths — bulk and CDC

Bulk

  • Partners who can only send full snapshots
  • SFTP via AWS Transfer Family → S3 → EMR Serverless / Glue → Iceberg bronze
  • Scales to 50M rows/hour
  • Reconciliation pain at scale

CDC preferred

  • Partners who give us DB access
  • Debezium on MSK Connect → Kafka → silver via Spark Structured Streaming
  • Sub-minute end-to-end freshness
  • Row-level deletes propagate correctly
  • No "missing row in snapshot" reconciliation

Contract clause: "You can start with bulk; we expect you on CDC inside 12 months." That's the right hill to die on.

What could go wrong — and how we handle it

If this happens……here's what we do
Partner sends the wrong file (last quarter's snapshot)Automated freshness + row-count checks → quarantine the file → alert the partner.
The AI mis-classifies a PII columnEvery AI suggestion is reviewed by a human first. AWS Macie also scans the raw landing zone for any PII the contract didn't declare.
Entity resolution merges two different peopleAuto-merge only above 95% confidence. Below that, a human reviews. Every merge is audited and reversible.
The PII vault has an outage during a bulk loadTokenization requests queue to a durable buffer. Bulk lags but doesn't fail.
Bedrock throttles us during a spikeReserved throughput keeps the hot path; the AI fallback handles the rest.
An engineer queries raw PII without a reasonVault requires a logged purpose + second-engineer approval. Every access lands in an audit feed.
Snowflake cost overrunsHard credit caps on every warehouse. Per-team query tagging. Weekly cost review.

Why Snowflake (instead of Redshift)

💸

Pay only when used

Spin up a warehouse for the nightly build, kill it after. We're not paying for idle compute.

🔒

Built-in PII protection

Per-role data masking is native — analysts see year-of-birth only, the sign-up service sees the full date. No app changes needed.

🔗

Share without copying

Partners can read a curated view of their own data inside their own warehouse. No S3 dumps, no nightly sync to maintain.

12 / 27

Why a PII vault (instead of just AWS encryption)

What AWS alone gives you

  • Encryption when data is at rest
  • Automated scans that find unexpected PII
  • Column-level access controls

Necessary — but not enough on its own.

What Skyflow adds

  • Lookalike tokens. Apps use a stand-in that has the same shape as the SSN — they never hold the real value.
  • Match without revealing. Find the right record without ever decrypting the PII.
  • Built-in approval flow. Anyone who wants to see the real value has to log a reason and get a second signoff.
  • Compliance certs included. SOC 2 + HIPAA + PCI in one signed agreement.
Buy the vault. Build the pipeline. Don't get those backwards.

14 / 27

Why Bedrock (vs. OpenAI direct)

  • Same VPC, no egress. Critical for HIPAA — model traffic never leaves AWS.
  • One BAA already in place at the AWS account level extends to Bedrock.
  • Model swap. Same SDK; switch Claude → Llama → Mistral without rewrite.
  • Provisioned throughput for predictable latency at high entity-res volume.
If we were a non-healthcare startup, OpenAI would be fine. We're in healthcare. Bedrock is the right answer.

AI feature #2 — Entity resolution

Are these two records the same person?

StageMechanismHandles
1 · Deterministic Exact match on tokenized SSN, or DOB + soundex(name) + ZIP5 ~70% — easy cases
2 · Embedding retrieval Bedrock Titan embeddings → OpenSearch k-NN top-K ~25% — typos, "Bob vs Robert", address moves
3 · LLM adjudication Claude scores each pair, returns JSON {decision, confidence, reasoning} borderline + audit trail

Confidence ≥ 0.95 → auto-merge. 0.80–0.95 → human review queue. < 0.80 → new golden record.

Pure rules miss 15–30% of true matches. Pure ML fails a HIPAA audit. Hybrid wins on every axis: precision, recall, auditability.

Who sees what — at the database

The same table, three different views — the database itself enforces it. Apps don't have to know.

Sign-up app

Sees full DOB, ZIP, last-4 SSN.

It needs them to verify identity.

Analyst / BI

Sees DOB year only. SSN columns return null.

Their dashboards still work — by year is plenty for cohort analysis.

Compliance

Sees everything, every access logged.

Audit trail with retention is HIPAA-compliant.

Three lines of SQL. Zero application changes. The database is the security boundary, not the app.
19 / 27

Detecting "is this the same person?"

We never trust a single rule. We triangulate two signals.

Hard signal

Same tokenized SSN

Tokens are unique per person, so if two records share one, they're the same person. Rare and certain.
→

Soft signal

Same DOB + name-sound + ZIP-prefix

Catches typos, marriage names, "Bob vs Robert". Imperfect — needs a second look.
Hard match → auto-merge · Soft match alone → human review queue · Neither → new record
Pure rules miss 15–30% of real matches. Pure ML fails a HIPAA audit. Two signals + a human-in-the-loop wins on every axis.

25 / 27

What I'd do differently in your real environment

  1. Talk to partners first. Half this design is informed by guesses. Two weeks reading existing code and looking at five real partner files would change details.
  2. Build the migration plan with operations. The interesting risk isn't building the new system. It's running both in parallel without breaking IDV for live members.
  3. Treat AI as augmentation, never a gate. When Claude returns garbage, a human gets paged with enough context to fix it in ten minutes. The schema-inference output is always reviewed before promotion. Always.

First 30 days: listening, reading, asking questions — not building.
First 90 days: the foundation phase from this plan, with the squad's input.
First 12 months: current state → target state with measurable improvements every quarter.
