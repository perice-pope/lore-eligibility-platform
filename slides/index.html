AI feature #2 — Entity resolution

Are these two records the same person?

StageMechanismHandles
1 · Deterministic Exact match on tokenized SSN, or DOB + soundex(name) + ZIP5 ~70% — easy cases
2 · Embedding retrieval Bedrock Titan embeddings → OpenSearch k-NN top-K ~25% — typos, "Bob vs Robert", address moves
3 · LLM adjudication Claude scores each pair, returns JSON {decision, confidence, reasoning} borderline + audit trail

Confidence ≥ 0.95 → auto-merge. 0.80–0.95 → human review queue. < 0.80 → new golden record.

Pure rules miss 15–30% of true matches. Pure ML fails a HIPAA audit. Hybrid wins on every axis: precision, recall, auditability.

Who sees what — at the database

The same table, three different views — the database itself enforces it. Apps don't have to know.

Sign-up app

Sees full DOB, ZIP, last-4 SSN.

It needs them to verify identity.

Analyst / BI

Sees DOB year only. SSN columns return null.

Their dashboards still work — by year is plenty for cohort analysis.

Compliance

Sees everything, every access logged.

Audit trail with retention is HIPAA-compliant.

Three lines of SQL. Zero application changes. The database is the security boundary, not the app.

The data contract — a translator the team controls

Acme HR system

Sends a CSV
"EmployeeID, FirstName, DOB (MM/DD/YYYY)…"
→

Data contract

One YAML file in git.
Versioned. Reviewed. Signed off.
→

Our pipeline

Sees one stable shape.
"partner_member_id, first_name, dob (ISO)…"

Each contract also declares PII tier per field, required completeness, cadence, retention, and auto-quarantine thresholds.

Partner changes their format? Bump the contract version, regression-test, deploy. No silent breakage.
